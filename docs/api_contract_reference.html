② 인증 계약

인증은 Authorization: Bearer <access_token> 헤더를 사용합니다.

  • 토큰이 없거나 유효하지 않으면 UNAUTHORIZED / 401.
  • 토큰의 type은 access여야 하며, 아니면 INVALID_TOKEN_TYPE / 401.
  • current_user_id는 actor_type=user, current_owner_id는 actor_type=owner만 허용합니다.
  • 로그인은 되어 있지만 actor가 맞지 않거나 소유권이 없으면 FORBIDDEN / 403.
  • 공개 조회 일부는 인증 없이 접근하며, 선택 인증은 user 토큰일 때만 viewer id를 사용합니다.
actor 범위: anonymous, user, owner, admin, system. 현재 API 의존성은 user/owner 중심입니다.

③ 페이지네이션(커서)

커서는 클라이언트가 해석하지 않는 opaque string입니다.

  • 요청 파라미터: cursor optional, limit 기본 20, 허용 범위 1..50.
  • 첫 요청은 cursor 없이 호출하고, 다음 페이지는 응답의 next_cursor를 그대로 전달합니다.
  • 일반 리스트 응답은 data, page.next_cursor, page.has_next, request_id 형태입니다.
  • 일부 피드 응답은 items, next_cursor 형태의 PageResult를 사용합니다.
  • 잘못된 cursor는 INVALID_CURSOR / 400.

④ Idempotency-Key

모든 변이 요청은 Idempotency-Key 헤더가 필요합니다.

  • 대상: 상태를 바꾸는 POST/PATCH/PUT/DELETE 계열 요청.
  • 누락 또는 빈 값이면 IDEMPOTENCY_KEY_REQUIRED / 400.
  • 같은 actor, 같은 key, 같은 method/path/body는 저장된 응답을 재사용합니다.
  • 같은 key로 다른 요청 본문을 보내면 IDEMPOTENCY_MISMATCH / 409.
  • 클라이언트는 재시도 가능한 요청마다 UUID 등 충분히 고유한 key를 생성해야 합니다.
