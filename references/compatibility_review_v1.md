## 1. 가장 시급한 충돌

### 🔴 1.1 웹푸시(Web Push) 채택 여부

| 출처 | 입장 |
|---|---|
| 사장님 웹앱 §9-1, §9-2 | "카카오 알림톡 + 웹 브라우저 푸시(Web Push)". 모든 알림 트리거에 둘 다 사용. 권한 요청 1회 명시 |
| 백엔드 §14 의사결정 / §13 알림 | "사장님 알림 방식: 카카오 알림톡, **웹푸시 MVP 제외**" (확정 결정사항으로 기록됨) |

**판단:** 백엔드 측은 이미 "확정"으로 분류한 항목인데 사장님 웹앱 spec은 정반대로 가정하고 있음. 둘 중 하나는 v1 단계에서 정정 필요.

### 🔴 1.2 계좌이체 안내 데이터 모델

| 출처 | 모델 |
|---|---|
| 사장님 웹앱 §2-3 | "은행명 / 계좌번호 / 예금주" **3개 필드 분리 입력** |
| 백엔드 Shop entity (`bank_transfer_guide_text`) | **단일 자유 text 필드** ("국민은행 123-45-678 (예금주: 홍길동)\n방문 전 입금 부탁드립니다" 형태 그대로 노출) |

**판단:** 입력 UI는 사장님 웹앱이 구조화된 폼을 가정, 백엔드는 자유 텍스트 1필드를 가정. 양쪽이 합치하지 않으면 사장님 웹이 3필드 폼을 노출했다 저장 시점에 single text로 직렬화하거나, 백엔드 모델을 분리 필드로 바꿔야 함. §14 미결정 항목 "계좌이체 안내 문구 입력 검증"과 직접 연관.

### 🔴 1.3 "예약금" 용어

| 출처 | 용어 |
|---|---|
| 사장님 웹앱 §2-3 | "**예약금** 결제수단 선택: 현장결제 / 계좌이체" |
| 백엔드 §14 의사결정 ("MVP 예약 결제") | "MVP 앱은 결제 시스템(PG·환불·**예약금**)을 구현하지 않는다... 앱은 안내 문구만 노출하고 입금 여부는 검증하지 않는다" |

**판단:** 백엔드는 "예약금" 개념 자체를 채택하지 않음. payment_method는 "결제 안내 방식"이지 "예약금 결제수단"이 아님. 사장님 웹앱 UI 카피가 "예약금"이라는 단어를 쓰면 유저/사장님 모두 혼선. 카피 정리 필요.

## 2. 백엔드 spec에는 있는데 사장님 웹앱이 누락한 항목

### 🟠 2.1 `owner_tags` 입력 UI (가장 중요)

- 백엔드 §6 / §14: 디자인 등록 시 **owner_tags 최소 1개 필수**, 0개면 400. AI 태그(`ai_tags`)와 별도 필드.
- 사장님 웹앱 §4-1 등록 입력 항목: 제목, 설명, 가격, 소요시간, 시술 가능 디자이너만 명시. **owner_tags 입력 UI 자체가 누락**

**판단:** 디자인 노출 조건이 `owner_tags >= 1 AND ai_analysis_status = done` 두 가지 모두 충족이라 백엔드의 핵심 정책. 사장님 웹앱 §4-1에 owner_tags 입력 필드와 안내(최소 1개 강제)를 반드시 추가해야 함.

### 🟠 2.2 LLM 재분석 동선 (사장님 웹앱은 "미정"으로 두지만 백엔드는 이미 결정)

- 백엔드 §6: `POST /owner/designs/{design_id}/reanalyze` API 이미 정의. `ai_analysis_status=failed` 시 또는 수동 트리거 가능.
- 사장님 웹앱 §4-2: "⚠️ 미정: 재분석 요청 / 이미지 재업로드 버튼 제공 여부"

**판단:** 백엔드는 결정·구현 방향까지 잡혀 있는데 사장님 웹앱은 미정. 사장님 웹앱 spec을 "재분석 버튼 제공" 으로 확정하고 §10 미정 표에서 제거 권장.

### 🟠 2.3 "[입금 완료] 알림" 사장님 측 동선

- 백엔드 §13 / §14: 유저가 "[입금 완료]" 누르면 사장님에게 카카오 알림톡 발송, 사장님 웹에는 "입금 확인 요청 표시" (`reservation_state_transitions` 의 `owner_web_display` 컬럼).
- 사장님 웹앱 §5-4: 사장님이 직접 통장 확인하는 흐름은 적혀 있으나, **사장님 웹 화면 어디에서 "입금 확인 요청" 상태가 노출되는지 정의 없음**. §9-2 알림 트리거 표에도 "유저 입금 완료 알림" 항목 누락.

**판단:** 사장님 웹앱 §5 또는 §9-2에 "유저 [입금 완료] 신호 수신 → 알림 + 예약 상세에 입금 확인 요청 표시" 항목 추가 필요.

### 🟠 2.4 "사장님 응답 리마인드 알림"

- 백엔드 §13 / §14: "pending이 N시간 이상 방치 시 사장님에게 카카오 알림톡 1회 발송" (N은 미정이지만 트리거 자체는 확정).
- 사장님 웹앱 §9-2 알림 트리거 표에 누락.

### 🟠 2.5 자동수락 샵 동선

- 백엔드: `auto_accept=true`면 예약 즉시 `confirmed`.
- 사장님 웹앱 §5-4 제목이 "**수동수락 샵**" 으로 한정. 자동수락 샵에서 사장님 측 동선(어떤 알림이 오는지, 캘린더에 어떻게 뜨는지)이 정의되어 있지 않음.

### 🟠 2.6 사업자 인증 대기 중 API 게이팅

- 사장님 웹앱 §1-2: "인증 대기 중 샵 등록/디자인 등록/예약 관리 등 운영 기능 **비활성화**".
- 백엔드 §5 owner API들: `verification_status=approved` 가드 정책이 spec 텍스트에 명시 없음. §16 권한 매트릭스도 owner의 verification 상태별 분기 없음.

**판단:** 백엔드 §5 또는 §16에 "verification_status != approved 시 owner_shop/owner_designer/owner_design/owner_reservation 계열 쓰기 API는 403" 정책 추가 필요.

## 3. 사장님 웹앱이 가정하는데 백엔드 spec이 누락한 항목

### 🟠 3.1 샵 다중 이미지 모델

- 사장님 웹앱 §2-1: "대표 이미지 + 샵 이미지 **복수 업로드**".
- 백엔드 Shop entity: `thumbnail_url` 단일 필드만 정의. `PATCH /owner/shops/{shop_id}/images` 요청 파라미터로 `image_urls` 가 잠깐 등장하지만 **엔티티에는 `image_urls` 컬럼/`ShopImage` 별도 테이블 정의 없음**.

**판단:** 백엔드 §5에 Shop.image_urls (URL[]) 또는 ShopImage 별도 엔티티 추가 필요.

### 🟠 3.2 예약 내역 CSV 다운로드 API

- 사장님 웹앱 §5-9: "취소/완료 처리된 예약 내역 CSV 다운로드 가능".
- 백엔드 §7: `GET /owner/reservations` (캘린더용 JSON) 만 있고 CSV/export 엔드포인트 없음.

**판단:** `GET /owner/reservations/export?format=csv&from=&to=` 류 엔드포인트 추가 필요.

### 🟠 3.3 리뷰 답변 수정/삭제 API

- 사장님 웹앱 §6-2: "답변 작성 후 **수정 / 삭제** 모두 가능".
- 백엔드 §10: `POST /reviews/{id}/reply` (생성) 만 정의. PATCH/DELETE 누락.

**판단:** `PATCH /reviews/{id}/reply`, `DELETE /reviews/{id}/reply` 추가 또는 reply를 독립 리소스로 분리.

### 🟠 3.4 사장님 알림 내역 API / 알림 엔티티

- 사장님 웹앱 §9-3: "웹앱 내 알림 목록", "읽음 / 안읽음 구분 표시".
- 백엔드 spec: Notification 엔티티 및 `GET /owner/notifications`, `PATCH /owner/notifications/{id}/read` 류 API 모두 없음.

**판단:** 백엔드에 알림 영속화 모델(NotificationDelivery 등) 및 사장님 조회/읽음 처리 API 추가 필요.

### 🟠 3.5 대시보드 집계

- 사장님 웹앱 §8: 오늘 예약 수, 신규 예약 요청 수, 미답변 리뷰 수, 최근 스네일 태그 수.
- 백엔드 spec: 통합 대시보드 API 없음. 각 리스트 API로 사장님 웹이 호출-필터-카운트 해야 함.

**판단:** `GET /owner/dashboard/summary` 류로 묶을지, 사장님 웹이 4개 호출을 동시에 날릴지 정책 결정 필요.

### 🟠 3.6 이미지 변경 시 LLM 재분석 자동 트리거

- 사장님 웹앱 §4-3: "이미지 변경 시 LLM 재분석 **자동 실행**".
- 백엔드 §6: `POST /owner/designs/{id}/images` / `DELETE /.../images/{image_id}` 만 있고, 자동으로 `ai_analysis_status=pending` 되돌리는 규칙 spec 텍스트에 명시 없음.

**판단:** 백엔드 §6 design entity 설명 또는 정책 섹션에 "이미지 추가/삭제 시 ai_analysis_status를 pending으로 되돌리고 큐 재투입" 명시 필요.

### 🟠 3.7 디자이너 표준 태그 사전 범위

- 사장님 웹앱 §3-1: 디자이너 전문 태그는 "표준 태그 사전 내에서 선택".
- 백엔드 Designer.specialty_tags는 string[] 선택. §12 LLM 명세에 standard_tags가 정의되어 있으나 디자이너 전문 태그가 그 중 어느 카테고리(style/technique/...)에서 가져오는지 미명시.

## 4. AI 분석 상태값 — 부분 불일치

### 🟡 4.1 표시 vs 저장 상태값 매핑

| 사장님 웹앱 §4-2 표 | 백엔드 `Design.ai_analysis_status` |
|---|---|
| 분석 중 | `pending` + `in_progress` (2개) |
| 분석 완료 | `done` |
| 분석 실패 | `failed` |

**판단:** 백엔드는 4개 상태, 사장님 웹앱은 3개 상태로 표시. 큐 대기(`pending`) vs 처리 중(`in_progress`)을 사장님 화면에서 한 덩어리로 묶을지 분리할지 결정 필요. §4-2 미정 항목 "분석 중/실패 표시 방식"과 연결됨.

## 6. 일치하는 항목 (참고용)

- 🟢 회원가입 입력 항목 (email/password/owner_name/phone): §1-1 ↔ Owner entity 일치
- 🟢 사업자 인증 흐름 (pending/approved/rejected): §1-2 ↔ Owner.verification_status 일치
- 🟢 1사장님=1샵 구조: §2-1 ↔ Shop.owner_id 단일 소속
- 🟢 디자이너 입력 항목 (이름/직급/경력/프로필/전문태그): §3-1 ↔ Designer entity 일치 (소개글 없음 명시도 일치)
- 🟢 디자이너 비활성화 처리: §3-4 ↔ Designer.is_active + 기존 예약 유지 정책 일치
- 🟢 디자인 이미지 최대 5장, 첫 장 썸네일: §4-1 ↔ DesignImage.sort_order(0=썸네일) + 5장 제한 일치
- 🟢 LLM 2단계(Transform → Classify) + 백그라운드 큐: §4-2 ↔ §12 일치
- 🟢 LLM 에러 코드 6종 (NO_NAIL/LOW_QUALITY/MULTIPLE_HANDS/OBSTRUCTED/INAPPROPRIATE/INTERNAL_ERROR): §4-2 ↔ §12 일치
- 🟢 예약 상태값 7종 (pending/confirmed/rejected/cancelled_by_user/cancelled_by_shop/no_show/completed): §5-2 ↔ §4 일치
- 🟢 노쇼 처리 30분 룰: §5-8 ↔ §4 state_transitions 일치
- 🟢 노쇼 시 앱 내 금전 처리 없음, 상태만 기록: §5-8 ↔ §14 일치
- 🟢 리뷰당 답변 1개 (생성): §6-2 ↔ Review.shop_reply 단일 필드 일치
- 🟢 스네일 댓글 - 샵 계정 가능 + 뱃지: §7-2 ↔ Comment.author_type=shop 일치
- 🟢 사장님 알림 = 카카오 알림톡 (웹푸시 제외하면): §9 ↔ §13 / §14 일치

## 7. 권장 후속 액션

### 사장님 웹앱 spec 측 갱신 필요

1. §4-1 디자인 등록에 **owner_tags 입력 UI(최소 1개 필수)** 추가
2. §4-2 미정 항목 "재분석 요청 / 이미지 재업로드 버튼" → "재분석 버튼 제공" 으로 확정
3. §5-4를 "수동수락 샵" → "수동수락 / 자동수락 분기" 로 확장, 자동수락 동선 명시
4. §5에 "유저 [입금 완료] 신호 수신 시 UI" 항목 신설
5. §9-2 알림 트리거 표에 "유저 [입금 완료] 알림", "사장님 응답 리마인드" 추가
6. §9 전반: **웹푸시 채택 여부를 백엔드와 합의 후 결정** (가장 시급)
7. §2-3 카피 정리: "예약금 결제수단" → "결제 안내 방식" 등 백엔드 모델과 정합

### 백엔드 spec 측 갱신 필요

1. §5 Shop entity에 **image_urls 또는 ShopImage 별도 엔티티** 추가
2. §5 또는 §16에 **사업자 인증(verification_status=approved) 전 owner 쓰기 API 게이팅** 정책 추가
3. §7에 **예약 CSV export 엔드포인트** 추가
4. §10에 **리뷰 답변 PATCH/DELETE** 또는 reply 독립 리소스 분리
5. **Notification 엔티티 + 사장님 알림 조회/읽음 처리 API** 추가
6. §6 design 정책에 **이미지 추가/삭제 시 ai_analysis_status pending 복귀 규칙** 명시
7. 대시보드 집계 API 정책 결정 (`/owner/dashboard/summary` vs 클라이언트 집계)
8. §2-3 / §5 계좌이체: **bank_transfer_guide_text 자유 텍스트 유지** vs **bank/account_number/account_holder 분리 필드** 합의

### 양쪽 합의 필요한 단일 결정

1. **웹푸시 채택 여부** (🔴 최우선)
