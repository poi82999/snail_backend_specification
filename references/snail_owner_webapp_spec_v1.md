# **1. 회원가입 / 로그인**

## **1-1. 회원가입**

- 입력 항목: 이메일, 비밀번호, 대표자명, 연락처

- **비밀번호 정책**: 8자 이상 + 영문/숫자 조합 1개 이상 (특수문자 강제 안 함)

- 가입 완료 후 사업자 인증 제출 화면으로 자동 이동

- **비밀번호 찾기/재설정**: 로그인 화면 [비밀번호 재설정] 링크 → 이메일 입력 → 백엔드가 토큰 이메일 발송(15분 TTL) → 링크 클릭하면 신규 비번 설정 화면. 미가입 이메일이어도 동일 화면 노출 (사용자 enumeration 방지)

## **1-2. 사업자 인증**

- 입력 항목: 사업자등록번호, 사업자등록증 이미지 또는 PDF 업로드

- 제출 후 상태: pending → 어드민 수동 검토

- 승인 완료: approved → 샵/디자인 공개 및 예약 운영 기능 활성화

- 반려: rejected → 반려 사유 화면에 표시, 재제출 가능

- 인증 대기 중에도 웹앱 탐색(읽기)과 샵/디자이너/디자인 초안 작성 가능

- 인증 대기 중 공개 전환, 유저 예약 접수, 예약 수락·거절·취소·완료·노쇼 등 운영 기능은 비활성화

- **백엔드 게이트**: 사장님 웹 UI에서 버튼을 숨기는 것뿐 아니라, 백엔드도 verification_status != approved인 경우 공개/예약 운영성 API에 대해 HTTP 403 + `VERIFICATION_REQUIRED` 코드를 반환 (UI 우회 방지). 게이트 적용 대상은 공개 전환 및 예약 수락·거절·취소·완료·노쇼·입금확인 API
  - 허용되는 API (인증 전이라도 호출 가능): 로그인/로그아웃, 자기 정보 조회·수정, 사업자 인증 제출, 알림함 조회, 공개 검색/탐색, 샵/디자이너/디자인 초안 작성
  - 프론트는 `FORBIDDEN`(소유권 위반)과 `VERIFICATION_REQUIRED`(인증 미완료)를 구분해 다른 안내 화면으로 유도

- **반려(rejected) 상태**: 반려 사유 화면에 표시 + 같은 사업자 인증 제출 API로 재제출 가능 (재제출 시 자동으로 pending 상태로 되돌아감)

## **1-3. 로그인**

- 이메일 + 비밀번호 로그인

- 로그인 실패 시 에러 문구 표시

- **실패 횟수 제한**: 5회 연속 실패 시 5분 잠금. 잠금 동안에는 "잠시 후 다시 시도해주세요" 안내. 잠금 해제 후 카운터 리셋

- 로그아웃: 클라이언트 토큰 삭제 + 서버 단기 블랙리스트 처리

## **1-4. 로그아웃**

- 로그아웃 시 세션 종료 후 로그인 화면으로 이동

# **2. 샵 설정**

## **2-1. 샵 기본 정보 등록**

- 1사장님 = 1샵 구조

- 입력 항목: 샵 이름, 주소, 상세주소, 전화번호, 샵 소개

- 주소 입력: 도로명 주소 API 연동 (MVP 포함)

- 대표 이미지 + 샵 이미지 복수 업로드 가능

## **2-2. 영업시간 설정**

- 요일별 오픈/마감 시간 설정

- 요일별 정기 휴무 설정 가능 (예: 매주 일요일 휴무)

- 특정 날짜 임시휴무는 디자이너 스케줄에서 개별 관리 (샵 단위 임시휴무 MVP 제외)

## **2-3. 예약 운영 설정**

- 예약 수락 방식 선택: 자동수락 / 수동수락

- 자동수락 기본값: false(수동수락)

- 예약 운영 안내 문구 직접 입력 (취소 / 노쇼 / 변경 안내)

- 예약금 결제 안내 방식 선택: 현장결제 / 계좌이체

- **현장결제**: 예약금 없음. 시술 당일 전액 결제 (앱 내 입금 안내/[입금 완료] 버튼 미노출)

- **계좌이체**: 샵 단위 고정 예약금 운용. 다음 4개 항목 입력 필수
  - 예약금 금액(원) — 모든 디자인에 동일 적용되는 정액
  - 은행명
  - 계좌번호
  - 예금주

- **조합 제약 (자동수락 ⟹ 현장결제 강제)**:
  - 자동수락 토글이 ON이면 결제 방식 선택지에서 **계좌이체 비활성화** (자동수락은 사장님이 매번 관여하지 않는 모드라 입금 모니터링이 어렵기 때문)
  - 자동수락 + 현장결제 / 수동수락 + 현장결제 / 수동수락 + 계좌이체 3개 조합만 허용
  - 사장님이 계좌이체로 운영 중인 샵을 자동수락으로 바꾸려 하면 안내 모달: "자동수락으로 전환하려면 현장결제로 먼저 변경하세요"
  - 백엔드도 VALIDATION_ERROR로 거부 (UI 우회 방지)

- ⚠️ 미정: 예약금 권장 범위/상한, 0원 입력 허용 여부

## **2-4. 샵 정보 수정**

- 기본 정보, 영업시간, 예약 운영 설정 각각 수정 가능

- 대표 이미지 / 샵 이미지 교체 가능

# **3. 디자이너 관리**

## **3-1. 디자이너 추가**

- 입력 항목: 이름, 직급(원장 / 실장 / 주니어 등), 경력(년수), 프로필 사진, 전문 태그

- 전문 태그는 MVP 프론트 입력 UI에서 제외 가능. 백엔드에는 선택 필드로만 유지

- 소개글 없음

## **3-2. 디자이너 정보 수정**

- 이름, 직급, 경력, 프로필 사진, 전문 태그 수정 가능

- 활성 / 비활성 상태 변경 가능

## **3-3. 주간 근무시간 설정**

- 디자이너별 요일별 근무 시작 / 종료 시간 설정

- 특정 날짜 휴무 또는 임시 불가 시간 등록 (날짜 + 시간대 + 사유)

## **3-4. 디자이너 비활성화 (퇴사)**

- 비활성화 처리 시 신규 예약 배정 후보에서 제외

- 기존 confirmed 예약은 유지됨

- 사장님이 해당 예약을 수동으로 취소 또는 처리

- 비활성화된 디자이너는 목록에서 구분 표시 (예: '비활성' 뱃지)

# **4. 디자인 등록 및 관리**

## **4-1. 디자인 등록 (사장님 작업)**

- 입력 항목: 디자인 제목, 설명, 가격(단일 입력), 소요시간(분), 시술 가능 디자이너 선택(복수), **사장님 태그(owner_tags, 0~N개 자유 입력 — 선택)**

- **owner_tags 입력 UI**: 칩 입력 형태 (Enter/콤마로 추가, X로 삭제). 안내 문구: "유저가 이 디자인을 검색할 때 잡힐 키워드. 비워둬도 등록·노출 가능하지만 입력하면 검색에 더 잘 잡힙니다 (AI 태그보다 가중치 높음)"

- 이미지 업로드: 최대 5장, 첫 번째 이미지가 썸네일. 백엔드도 디자인당 총 5장 제한을 API에서 검증

- 등록 완료 즉시 DB 저장 → 사장님 작업 끝

- 사장님 웹에는 '등록 완료' 상태로 표시

## **4-2. LLM 분석 (백그라운드 자동 처리)**

- 백엔드가 등록된 디자인을 Queue에 순차 적재

- 1단계 Transform: 원본 이미지에서 네일 영역 추출 및 규격화

- 2단계 Classify: 태그 / 색상 / 분위기 / 스타일 자동 분류

- 분석 완료된 디자인만 유저 앱에 노출

**LLM 분석 상태 (사장님 화면 표시 단위 — 3단계)**

| **사장님 화면 표시** | **백엔드 ai_analysis_status** | **설명** | **앱 노출** |
| --- | --- | --- | --- |
| 분석 중 | pending + in_progress | Queue 대기 + LLM 처리 중을 묶어 한 덩어리로 표시 | ❌ |
| 분석 완료 | done | 태그/색상/스타일 분류 완료 | ✅ |
| 분석 실패 | failed | 에러 코드와 함께 실패 사유 표시. [재분석] 버튼 노출 | ❌ |

**LLM 분석 실패 에러 코드**

| **에러 코드** | **설명** | **사장님 안내 문구** |
| --- | --- | --- |
| NO_NAIL | 사진에서 네일 감지 안 됨 | 손이 잘 보이는 사진으로 다시 올려주세요. |
| LOW_QUALITY | 해상도 낮음 / 블러 / 조명 부족 | 좀 더 선명한 사진으로 시도해주세요. |
| MULTIPLE_HANDS | 여러 명의 손이 섞임 | 한 사람의 손만 나오는 사진으로 올려주세요. |
| OBSTRUCTED | 네일이 가려짐 | 네일이 잘 보이도록 다시 촬영해주세요. |
| INAPPROPRIATE | 부적절 콘텐츠 감지 | 이 사진은 업로드할 수 없습니다. |
| INTERNAL_ERROR | LLM 내부 에러 | 잠시 후 다시 시도해주세요. |

- ⚠️ 미정: 분석 중 / 실패 상태 사장님 웹 표시 방식 (진행바 vs 뱃지 등)

- **재분석 버튼 제공**: `ai_analysis_status=failed`인 디자인 상세에 [재분석] 버튼 노출. 클릭 시 `POST /owner/designs/{design_id}/reanalyze` 호출 → 상태 `pending`으로 되돌리고 큐 재투입. 사장님이 수동으로 다시 돌리고 싶을 때도 동일 버튼 사용 가능

## **4-3. 디자인 수정**

- 제목, 설명, 가격, 소요시간, 시술 가능 디자이너 수정 가능

- 이미지 추가 / 삭제 가능 (최대 5장 유지)

- 이미지 변경 시 LLM 재분석 자동 실행 (백엔드가 ai_analysis_status를 pending으로 되돌리고 큐 재투입)

## **4-4. 디자인 노출 / 숨김**

- 사장님이 개별 디자인 노출 / 숨김 전환 가능

- 숨김 처리된 디자인은 앱에 노출되지 않음

- 분석 중 또는 분석 실패 상태는 자동 미노출

## **4-5. 디자인 삭제**

- 삭제 시 확인 모달 표시

- **확정 예약이 있는 디자인 삭제**: hard delete 안 함. 백엔드가 자동으로 visibility=hidden + soft delete 처리 → 신규 노출 차단, 기존 confirmed 예약은 그대로 유지. 관련 예약이 모두 완료/취소된 후 30일 뒤 hard delete

- 삭제 시 사장님에게 "현재 X건의 확정 예약이 있는 디자인입니다. 삭제하면 신규 노출은 중단되지만 기존 예약은 정상 진행됩니다" 안내 모달 권장

# **5. 예약 관리**

## **5-1. 예약 캘린더**

- **MVP 기본 뷰: 월간만** (일간/주간 뷰 전환은 post-MVP)

- 디자이너별 필터링 가능

- 예약 상태별 색상 구분 표시

- 캘린더에서 예약 클릭 시 예약 상세로 이동

- 미처리 pending 예약은 별도 목록으로 created_at 오름차순(먼저 예약 요청한 순서) 표시. 같은 슬롯에 여러 pending 요청이 있으면 사장님이 목록 순서대로 응답하도록 유도

- 백엔드 GET /owner/reservations는 from/to 임의 범위 지원하므로 추후 일간/주간 추가 시 백엔드 변경 없음

## **5-2. 예약 상태 정의**

| **상태값** | **설명** | **유저 앱 표시** | **사장님 웹 표시** |
| --- | --- | --- | --- |
| pending | 수락 대기 | 승인 대기 중 | 신규 예약 |
| payment_pending | 입금 확인 대기 | 입금 확인 대기 | 입금 확인 대기 |
| confirmed | 확정 | 예약 완료 | 확정 예약 |
| rejected | 거절됨 | 거절됨 | 거절 예약 |
| cancelled_by_user | 유저 취소 | 취소됨 | 유저 취소 |
| cancelled_by_shop | 샵 취소 | 샵 사정으로 취소됨 | 샵 취소 |
| no_show | 노쇼 | 노쇼 처리됨 | 노쇼 |
| completed | 완료 | 리뷰 작성 유도 | 완료 |

## **5-3. 예약 상세 확인**

- 표시 항목: 유저 정보, 디자인, 예약 시간, 담당 디자이너, 유저 요청사항, 예약 상태

- 예약 운영 안내(취소 / 노쇼 정책) 스냅샷 확인 가능

- **입금 확인 요청 뱃지 (계좌이체 샵 한정)**:
  - 표시 조건: `status=payment_pending` AND `payment_method_snapshot=bank_transfer_guide` AND `user_payment_notified_at != null` AND `owner_payment_confirmed_at IS NULL`
  - 즉 유저가 [입금 완료]를 눌렀고 사장님이 아직 처리 안 한 경우
  - 캘린더에서도 해당 예약 칸에 시각 마커 표시 (색상 또는 작은 아이콘)
  - 뱃지 클릭 시 모달 표시 — 두 가지 선택지
    - **[입금 확인됨]** → `POST /owner/reservations/{id}/payment-confirmed` 호출 → `owner_payment_confirmed_at` 기록 → `confirmed` 전환 → 뱃지 사라짐
    - **[샵 사정 취소]** (미입금 시) → 기존 `POST /owner/reservations/{id}/cancel` 호출 → cancelled_by_shop로 전환 → 뱃지 자연히 사라짐

## **5-4. 예약 수락 및 입금 안내**

### 수동수락 샵

- 신규 예약 요청 시 카카오 알림톡 + 사장님 웹 알림함 수신

- 사장님이 수락 버튼 클릭
  - 현장결제 샵: confirmed 상태로 변경
  - 계좌이체 샵: payment_pending 상태로 변경 + 유저에게 계좌/예약금 안내

- 유저에게 APNs 푸시 발송: 현장결제는 "사장님이 예약을 수락하셨어요!", 계좌이체는 "예약금 입금 후 입금 완료를 눌러주세요"

- 계좌이체 샵: 푸시에 예약금 금액 + 계좌 정보(은행명 / 계좌번호 / 예금주) 포함

- 현장결제 샵: 예약금 안내 미포함

- 유저가 입금 후 앱에서 [입금 완료] 버튼 직접 클릭 → 사장님에게 카카오 알림톡 + 알림함 적재 "예약 #XXX 유저가 입금 완료를 알렸어요"

- 사장님이 통장에서 입금 확인 후 [입금 확인됨] 클릭 → confirmed 상태로 변경

- 미입금 시 사장님이 예약 취소(cancelled_by_shop) 가능 (자동 만료 없음)

- 입금 여부는 사장님이 통장에서 직접 확인 (자동 검증 없음)

### 자동수락 샵

- 예약 요청 즉시 confirmed 상태로 생성

- 사장님에게는 "예약 자동 확정됨" 알림(카카오 알림톡 + 알림함) 1건만 발송

- 유저 측 푸시는 수동수락과 동일 (계좌이체 샵이면 예약금/계좌 안내 포함)

## **5-5. 예약 거절**

- 거절 버튼 클릭 → 거절 사유 입력 (필수)

- rejected 상태로 변경

- 유저에게 거절 알림 + 사유 전달

## **5-6. 샵 사정 취소**

- confirmed 예약을 사장님이 취소 가능

- **취소 사유 입력 필수** (텍스트) — 거절(reject)과 동일하게 필수

- cancelled_by_shop 상태로 변경

- 유저에게 취소 알림 + 사유 전달

## **5-7. 시술 완료 처리**

- 사장님이 수동으로 completed 처리

- completed 전환 시 유저 앱에 리뷰 작성 유도 APNs 알림 발송

## **5-8. 노쇼 처리**

- 시술 시작 시간 기준 30분 이후부터 노쇼 처리 버튼 활성화

- no_show 상태로 변경

- 앱 내 금전 처리 없이 상태 기록만 남김

- 유저 노쇼 이력 내부 기록

## **5-9. 예약 내역 다운로드**

- **MVP 1차 출시 제외. post-MVP 우선순위 큐로 분류**

- 사장님 실사용 피드백 받고 필요한 컬럼/기간/필터 구체화 후 추가 예정

- 추가 시 GET /owner/reservations/export?from=&to=&status=&format=csv 형태. from/to 최대 90일 범위 제한

# **9. 알림**

## **9-1. 알림 수신 방식**

- **카카오 알림톡**: 사장님 연락처로 발송 (웹앱 미접속 상태에서도 수신)

- **사장님 웹 알림함(in-app inbox)**: 모든 사장님 대상 알림이 영속화. 카톡을 놓쳐도 웹 접속 시 다시 확인 가능

- 웹 브라우저 푸시(Web Push)는 MVP에서 제외 (iOS Safari 제약 및 카톡과의 효용 중복)
