# 네일 예약 + 커뮤니티 플랫폼 백엔드 협업 명세서 v3 요약

이 문서는 엑셀을 직접 뒤지지 않기 위한 중간 분석/수정용 자료입니다. 앞으로 백엔드 협업명세서 관련 탐색은 이 문서를 우선 기준으로 합니다.

## 현재 명세의 우선 목표

이 명세서의 1차 목표는 백엔드 구현 세부사항을 모두 확정하는 것이 아닙니다. 프론트엔드 작업자, UI 디자이너, 사장님 웹 작업자, LLM 작업자가 백엔드 지식이 많지 않아도 각자 필요한 결정을 빠짐없이 채울 수 있게 만드는 것이 우선입니다.

따라서 각 협업자는 DB 내부 구조를 설계하기보다, 자신이 담당하는 화면/작업 흐름에서 다음 질문에 답하면 됩니다.

- 이 API는 어느 화면, 어느 버튼, 어느 상태에서 호출되는가.
- 성공했을 때 유저 또는 사장님에게 무엇을 보여주는가.
- 실패했을 때 어떤 문구와 재시도 동선을 보여주는가.
- 로딩 중, 빈 결과, 권한 없음, 중복 클릭 상황은 어떻게 처리하는가.
- 백엔드가 제안한 기본 정책 중 화면/UX 관점에서 어색한 부분이 있는가.

## 협업자별 작성 책임

- 프론트엔드 작업자: API 호출 시점, 요청 파라미터 수용 가능 여부, 응답 필드 필요 여부, 에러/빈 상태/로딩 상태 처리, 딥링크, 페이지네이션 처리 가능 여부를 확인한다.
- UI 디자이너: 예약 요청, 승인 대기, 예약 확정, 거절, 취소, 노쇼, 분석 실패, 권한 없음, 빈 결과 화면의 문구와 버튼 흐름을 확인한다.
- 사장님 웹 작업자: 샵/디자이너/디자인/예약 관리 화면에서 필요한 CRUD 흐름, 입력 UI, 검증 규칙, 사장님에게 보여줄 상태값을 확인한다.
- LLM 작업자: Transform/Classify API, callback, 재시도, 모델 버전, 이미지 저장 책임, 태그 사전, 에러 코드를 확정한다.
- 백엔드: DB 정합성, 권한, 상태 전이, 예약 충돌 방지, 공통 응답 포맷, 동시성 제어의 기본안을 제시하고 협업자가 이해할 수 있는 형태로 질문을 나눈다.

## 표를 읽는 법

각 표의 노란색 칸은 실제 운영 데이터를 적는 곳이 아니다. 협업자가 “화면/UX/API 연동 관점에서 이 항목을 어떻게 처리할지”를 적는 곳이다.

### 필드 표

- `백엔드 필드명`: 서버와 DB에서 쓰는 이름이다. 화면에 그대로 노출할 필요는 없다.
- `화면 라벨/표현`: 앱/웹 화면에서 유저에게 보여줄 이름이다. 협업자가 더 자연스러운 표현을 제안한다.
- `타입`: 서버가 기대하는 데이터 모양이다. 예를 들어 `int`는 숫자, `bool`은 true/false, `UUID`는 내부 ID다.
- `필수?`: 유저나 사장님이 반드시 입력해야 하는지, 서버가 자동 생성하는지 구분한다.
- `필드 의미`: 이 필드가 무엇을 뜻하고 어떤 업무 흐름과 연결되는지 설명한다.
- `입력/생성 방식`: 유저/사장님이 직접 입력하는지, 서버가 자동 생성하는지, 특정 조건에서만 필요한지 설명한다.
- `협업자 확인 사항`: 화면에서 보일지, 입력받을지, 수정 가능할지, 어떤 에러 문구가 필요한지 확인한다.
- `추가 요청/결정`: 화면 구현에 필요한 추가 응답 필드, 문구, 예외 처리, 빈 상태, 정렬/필터 필요 여부를 적는다.

협업자는 필드를 보고 “DB에 어떤 컬럼을 만들지”를 고민하지 않아도 된다. 대신 이 필드가 화면에서 보이는지, 입력받는지, 수정 가능한지, 실패 시 어떤 문구가 필요한지를 적으면 된다.

### API 표

- `엔드포인트`: 프론트/웹이 호출할 서버 주소다.
- `용도`: 이 API가 수행하는 사용자 동작이다.
- `요청값`: 프론트/웹이 서버에 보내야 하는 값이다.
- `호출 화면/상황`: 어느 화면, 버튼, 탭, 상태에서 호출되는지 적는다.
- `프론트 확인 사항`: 요청값을 준비할 수 있는지, 성공 후 어떤 화면으로 이동할지, 실패 문구가 필요한지 확인한다.
- `추가 요청/결정`: 응답에 더 필요한 필드, 상태값, 정렬/필터, 페이지네이션, 빈 상태 문구를 적는다.

## 페이지별 협업자 가이드

### 1.README

- 다루는 것: 문서 목적, 색상 규칙, 전체 시스템 구성, 협업자가 채워야 하는 범위.
- 관련 업무: 모든 협업자가 처음 읽는 안내 페이지.
- 협업자 관점: 내가 어느 시트를 봐야 하는지, 노란색 칸에 무엇을 써야 하는지 확인한다.

### 2.전체플로우

- 다루는 것: 사장님 등록, LLM 분석, 유저 검색/예약, 알림, 리뷰/스네일까지의 큰 흐름.
- 관련 업무: 프론트, 사장님 웹, LLM 작업자가 자신의 작업이 전체 흐름 어디에 들어가는지 이해하는 데 사용한다.
- 협업자 관점: 내가 만드는 화면이나 API 호출이 어느 단계에서 발생하는지 확인한다.

### 3.유저(앱)_회원

- 다루는 것: Apple Sign In, 유저 프로필, 닉네임, 프로필 사진, 관심 태그, APNs 토큰.
- 관련 업무: 앱 로그인/회원가입/프로필 수정 화면.
- 협업자 관점: 회원가입 시 꼭 필요한 입력값, 프로필 수정 가능 항목, 닉네임/소개글 검증 규칙을 채운다.

### 4.유저(앱)_탐색예약

- 다루는 것: 검색, 피드, 디자인 상세, 찜, 예약 요청, 예약 상태, 가용 시간.
- 관련 업무: 앱 홈, 검색 결과, 디자인 상세, 예약 시간 선택, 예약 요청, 내 예약 화면.
- 협업자 관점: 각 상태에서 어떤 버튼을 보여줄지, 예약 충돌/예약 취소/사장님 거절을 어떻게 보여줄지 확인한다.

### 5.사장님(웹)_샵관리

- 다루는 것: 사장님 계정, 사업자 인증, 샵 정보, 영업시간, 예약 운영 안내, 디자이너, 스케줄.
- 관련 업무: 사장님 웹 가입, 샵 설정, 디자이너 관리, 근무시간 관리.
- 협업자 관점: 사장님이 직접 입력하기 쉬운 UI인지, 사업자 승인 전 제한 화면이 필요한지, 영업시간/예약 운영 안내 입력 방식이 자연스러운지 확인한다.

### 6.사장님(웹)_디자인

- 다루는 것: 디자인 등록, 가격, 소요시간, 가능 디자이너, 이미지, LLM 분석 상태.
- 관련 업무: 사장님 웹 디자인 등록/수정/숨김/삭제, 이미지 업로드, LLM 재분석.
- 협업자 관점: 분석 중/실패/완료 상태를 어떻게 보여줄지, 디자인 삭제와 숨김을 어떻게 구분할지 확인한다.

### 7.사장님(웹)_예약

- 다루는 것: 예약 캘린더, 예약 상세, 수락/거절, 샵 취소, 완료, 노쇼.
- 관련 업무: 사장님 웹 예약 관리 화면.
- 협업자 관점: 예약 상태별 버튼, 거절/취소 사유 입력, 노쇼 처리 가능 시간, 캘린더 필터를 확인한다.

### 8.커뮤니티_스네일

- 다루는 것: 스네일 피드, 랭킹, 팔로잉, 스네일 작성, 샵/디자인/예약 태그.
- 관련 업무: 앱 커뮤니티 탭, 스네일 작성/상세 화면.
- 협업자 관점: 스네일 작성 시 태그 선택 UI, 인증 뱃지, 디자인 상세에 노출되는 조건을 확인한다.

### 9.커뮤니티_댓글

- 다루는 것: 댓글, 대댓글, 좋아요, 팔로우, 유저/샵 작성자 구분.
- 관련 업무: 스네일 상세 댓글 영역, 알림, 팔로우 기능.
- 협업자 관점: 댓글 depth 2 UI, 샵 댓글 뱃지, 삭제된 댓글 표시, 좋아요 상태 표시를 확인한다.

### 10.커뮤니티_리뷰

- 다루는 것: 예약 완료 후 리뷰, 샵 별점, 리뷰 사진, 샵 답변.
- 관련 업무: 리뷰 작성 화면, 디자인/샵 상세 리뷰 목록, 사장님 리뷰 답변.
- 협업자 관점: 리뷰 작성 가능 조건, 사진 5장 제한, 리뷰 수정/삭제 기간, 샵 답변 노출 방식을 확인한다.

### 11.커뮤니티_신고

- 다루는 것: 스네일/댓글/리뷰/유저/샵 신고와 운영 처리.
- 관련 업무: 신고 버튼, 신고 사유 선택, 운영자 모더레이션.
- 협업자 관점: 신고 사유 문구, 신고 완료 안내, 신고 후 콘텐츠 표시 여부를 확인한다.

### 12.LLM명세

- 다루는 것: 네일 이미지 Transform, 태그 Classification, 태그 사전, 에러 코드, callback/재시도/저장 책임.
- 관련 업무: LLM API 구현, 백엔드-LLM 연동, 디자인 등록 자동 분석.
- 협업자 관점: LLM 작업자는 노란색 답변 칸에 실제 API 운영값을 채우고, 프론트/웹은 분석 실패 문구와 재업로드 동선을 확인한다.

### 13.알림

- 다루는 것: 유저 APNs, 사장님 카카오 알림톡, 예약/리뷰/스네일/LLM 분석 알림.
- 관련 업무: 앱 푸시, 사장님 카톡 알림, 알림 딥링크.
- 협업자 관점: 알림 문구, 딥링크 목적지, 너무 잦은 알림을 묶을 기준을 확인한다.

### 14.의사결정기록

- 다루는 것: 이미 확정된 정책과 아직 팀 내부 결정이 필요한 항목.
- 관련 업무: 팀 회의, 정책 확정, 문서 버전 관리.
- 협업자 관점: 이미 결정된 내용은 다시 논쟁하지 않고, 미결정 항목만 회의에서 확정한다.

### 15.체크리스트

- 다루는 것: 담당자별 회신 전 점검 항목.
- 관련 업무: 협업자별 명세 검토 완료 확인.
- 협업자 관점: 자신의 담당 항목에 예/아니오/미정을 표시하고, 막히는 항목은 `메모/막힌 점`에 남긴다.

### 16.공통_API권한

- 다루는 것: 공통 응답 형식, 에러 코드, 페이지네이션, rate limit, 권한 표.
- 관련 업무: 모든 프론트 API 연동, 로그인 필요/권한 없음/빈 상태 처리.
- 협업자 관점: 모든 화면에서 같은 방식으로 에러와 리스트를 처리할 수 있는지 확인한다.

## 팀 내부 결정 필요 사항

아래 항목은 특정 협업자에게 바로 넘기기 전에 우리 팀 내부에서 먼저 방향을 정해야 합니다.

- 문서 버전 전략: v3를 계속 수정할지, v4 파일로 분리해 확정본을 만들지 결정.
- 향후 결제 기능 범위: 예약금/PG/환불 기능을 어느 버전에서 도입할지 결정. MVP 앱에서는 결제 기능을 구현하지 않는다.
- 예약 운영 안내 문구: 결제 없는 예약 요청 구조에서 유저에게 취소/노쇼/변경 안내를 어떻게 보여줄지 결정.
- 결제 없는 예약 보완 기준: 활성 예약 수 제한, 노쇼 이력 기록, 취소 이력 기록, 리마인드 알림, 수동수락 기본 운영 범위를 결정.
- 예약 자동수락 기본값: 샵 생성 시 `auto_accept` 기본값을 true/false 중 무엇으로 둘지 결정.
- 예약 슬롯 단위: 15분 단위와 30분 단위 중 MVP 기본값 결정.
- 예약 버퍼 시간: 시술 전후 준비/정리 시간을 둘지, 둔다면 샵/디자인/디자이너 중 어디에 설정할지 결정.
- 당일 예약 마감: 몇 시간 전까지 예약 가능한지 기본 정책 결정.
- 사장님 계정 인증: 이메일/비밀번호만 쓸지, 휴대폰 인증 또는 사업자 승인 전 기능 제한을 둘지 결정.
- 어드민 범위: MVP에서 실제 어드민을 만들지, 운영자가 DB/내부툴로 처리할지 결정.
- 공통 API 형식: 에러 응답, 페이지네이션, cursor, rate limit 문구의 기본안을 백엔드가 먼저 고정할지 결정.
- LLM 책임 경계: 이미지 저장, callback 인증, 재시도, 모델 버전 표기를 LLM 작업자가 제안하면 백엔드가 수용하는 방식으로 갈지 결정.

## v3 변경사항

- LLM 명세 강화: Visual Transformer 네일 추출 + Classification 태그 분류 2단계.
- 커뮤니티 구조 확정: 통합 커뮤니티 + 디자인 상세에서 해당 디자인의 스네일/리뷰 노출.
- 리뷰 정책 확정: 1예약=1리뷰, 샵 별점만, 디자이너 별점 없음, 사진 최대 5장, 작성 선택.
- 사장님 알림 방식 확정: 카카오 알림톡. 웹푸시는 MVP 제외.
- 디자이너 자동 배정: 유저가 미선택하면 가능한 디자이너 중 랜덤.
- 노쇼 정책: MVP에서는 앱 내 금전 처리 없이 사장님이 예약 상태를 `no_show`로 표시.
- 검색 폴백: 0건 시 유사 디자인 추천.
- 이달의 아트: MVP 범위에서 제외.
- 검색 방식: LLM 검색이 아니라 DB 기반 자연어형 검색으로 구현.

## MVP 방향성 확정

- 홈 상단 메뉴는 추천 / 네일샵 중심으로 시작한다.
- `이달의 아트`는 MVP에서 제외하고, 추후 큐레이션 영역으로 재검토한다.
- 검색은 LLM 질의 처리 없이 DB 기반 자연어형 검색으로 구현한다.
- 검색어는 자연어처럼 받을 수 있지만, 내부 처리는 정규화, 태그 사전, 동의어 사전, 오타 보정, DB 필터 검색을 사용한다.
- 날짜 조건은 검색에서 제외하고, 디자인 상세 또는 예약 화면에서 실제 가용 시간을 조회한다.
- MVP 앱에서는 결제 기능을 구현하지 않는다.
- 결제 없는 예약은 활성 예약 수 제한, 노쇼/취소 이력, 예약 리마인드, 사장님 수동수락으로 보완한다.
- 자동 배정은 유지하되 MVP에서는 가능한 디자이너 중 랜덤 배정 수준으로만 다룬다.
- `스네일`은 무신사 SNAP처럼 스타일 공유 커뮤니티로 정의한다.
- 리뷰는 예약 완료 후 작성하는 평가 데이터로 정의한다.
- 디자인 상세에서는 스네일과 리뷰를 분리해서 보여준다.

## 시스템 주체

- 유저 앱: iOS/TestFlight. 검색, 예약, 스네일, 좋아요, 댓글, 리뷰.
- 사장님 웹: 반응형 웹. 샵, 디자이너, 디자인, 예약 관리.
- LLM 서비스: 백엔드가 호출. 1단계 Transform, 2단계 Classify.
- 카카오 알림톡: 백엔드가 사장님 알림 발송.
- 백엔드: 두 클라이언트, LLM, 알림톡, DB/API 연결.
- 운영자/어드민: 신고, 사업자 승인, 모더레이션.

## 전체 플로우

1. 사장님이 회원가입, 샵 등록, 디자이너/근무시간을 입력한다.
2. 사장님이 디자인 사진, 가격, 소요시간, 가능 디자이너를 등록한다.
3. 백엔드는 원본 이미지를 저장하고 디자인을 `draft`로 생성한다.
4. 백엔드는 LLM 1단계 Transform을 호출해 네일 영역을 추출한다.
5. 백엔드는 LLM 2단계 Classification을 호출해 태그, 색상, 스타일을 받는다.
6. 디자인 상태를 `active`로 바꾸고 유저에게 노출한다.
7. 유저는 검색/피드/상세에서 디자인을 보고 예약한다.
8. 유저는 앱에서 결제 없이 예약 요청을 생성하고, 중복 요청 방지용 `Idempotency-Key`를 함께 보낸다.
9. 사장님은 카카오 알림톡을 받고 수락/거절한다.
10. 예약 완료 후 유저는 선택적으로 리뷰를 작성한다.
11. 유저는 자유롭게 스네일을 올릴 수 있고, 디자인을 태그하면 디자인 상세에 노출된다.

## 핵심 정책

### 검색/탐색

- MVP 상단 홈 메뉴는 추천 / 네일샵 중심으로 구성한다. `이달의 아트`는 MVP 범위에서 제외한다.
- 통합 검색 결과 범위: 디자인, 샵, 리뷰.
- 검색어는 유저가 자연어처럼 입력할 수 있지만, 백엔드는 LLM이 아니라 DB 기반 검색으로 처리한다.
- 백엔드는 검색어를 정규화하고 태그/동의어/오타 보정 사전으로 매칭한다.
- 예: “여리여리한 핑크 네일” → `핑크`, `러블리`, `내추럴`, `심플` 같은 태그 후보와 매칭.
- 날짜 조건은 검색 API에서 제외한다. 실제 예약 가능 시간은 디자인 상세 또는 예약 화면에서 조회한다.
- 찜한 사람 수를 카드에 표시한다.
- 디자이너 단독 검색은 제외한다.
- 검색 결과 0건이면 빈 화면 대신 유사 태그 기반 추천 디자인을 반환한다.
- 디자인 상세는 디자인 정보, 해당 디자인을 받은 스네일, 해당 디자인의 리뷰를 함께 반환한다.

MVP 검색 구현 기본안:

- `q`: 자유 검색어. 자연어처럼 입력 가능.
- `region`: 지역 또는 좌표 기반 지역 필터.
- `duration_min/duration_max`: 소요시간 필터.
- `price_min/price_max`: 가격 필터.
- `colors[]`: 색상 태그 필터.
- `moods[]`: 분위기 태그 필터.
- `cursor/limit`: 페이지네이션.
- 내부 처리: 검색어 정규화 → 태그/동의어 매칭 → 오타 보정 → DB 검색 → 유사 결과 fallback.

### 예약

- 디자이너 지정 예약과 자동 배정 예약을 모두 지원한다.
- 자동 배정은 선택 시간에 가능한 디자이너 중 랜덤이다.
- MVP 앱에서는 예약금 결제 기능을 구현하지 않는다.
- 예약은 결제 없이 “예약 요청”으로 생성되고, 사장님 수락 또는 샵 자동수락에 따라 확정된다.
- 취소/거절/노쇼는 앱 내 금전 처리 없이 예약 상태로만 관리한다.
- 결제 없는 예약의 노쇼 리스크는 활성 예약 수 제한, 노쇼/취소 이력 기록, 예약 전 리마인드, 사장님 수동수락으로 보완한다.
- 예약 생성에는 `Idempotency-Key`가 필수다.
- 동시성은 DB UNIQUE + 트랜잭션으로 막고 충돌 시 409를 반환한다.
- 예약 시점 가격은 `reservations.total_price`에 스냅샷으로 저장한다.

### 예약 엔티티 기본안

이 영역은 협업자가 DB를 설계하는 영역이 아니라, 백엔드가 기본안을 제공하고 프론트/UI가 화면 흐름을 확인하는 영역이다.

#### Reservation

- 예약의 본체다.
- 유저, 샵, 디자인, 배정 디자이너, 시작/종료 시간, 예약 상태, 예약 시점 가격, 예약 운영 안내 스냅샷을 가진다.
- 디자이너 자동 배정인 경우에도 최종 확정된 `designer_id`는 저장한다.
- `assigned_by`로 유저 선택인지 자동 배정인지 구분한다.
- 협업자 확인 사항: 예약 카드에 총액/디자이너 자동배정 여부/승인 대기 상태/사장님 확인 필요 여부를 어떻게 표시할지.

#### 향후 Payment / Refund

- MVP 앱에서는 구현하지 않는다.
- 예약금 결제, PG 연동, 환불 기록은 향후 기능으로 분리한다.
- 현재 문서에서는 예약 요청/승인/취소/노쇼 상태만 다룬다.

#### IdempotencyKey

- 예약 생성 요청의 중복 처리를 막기 위한 서버 내부 기록이다.
- 같은 키로 같은 요청이 반복되면 첫 요청 결과를 재사용한다.
- 다른 body로 같은 키가 오면 409 또는 422 계열 에러로 막는다.
- 협업자 확인 사항: 예약 요청 버튼 중복 클릭 방지 UI, 로딩 중 버튼 비활성화, 네트워크 재시도 방식.

#### 결제 없는 예약 보완 정책

- 유저는 로그인 상태에서만 예약 요청을 만들 수 있다.
- 유저당 동시에 유지할 수 있는 활성 예약 수를 제한한다. 예: `pending + confirmed` 최대 2~3개.
- 같은 샵/같은 날짜/비슷한 시간대의 중복 예약 요청을 제한한다.
- 사장님이 노쇼 처리한 횟수를 유저 내부 이력으로 기록한다.
- 잦은 취소 이력도 내부적으로 기록한다.
- 노쇼 또는 직전 취소가 누적된 유저는 예약 요청 제한 또는 경고 문구 노출을 고려한다.
- 예약 전날/당일 리마인드 푸시를 보낸다.
- MVP에서는 사장님 수동수락을 기본 운영으로 두고, 자동수락은 샵이 선택할 수 있게 한다.
- 예약 요청 전 유저에게 취소/노쇼 안내 문구를 보여준다.

### 예약 상태

- `pending`: 사장님 수락 대기.
- `confirmed`: 확정.
- `rejected`: 사장님 거절.
- `cancelled_by_user`: 유저 취소.
- `cancelled_by_shop`: 샵 취소.
- `no_show`: 노쇼.
- `completed`: 시술 완료.

### 예약 상태 전이표

상태 전이표는 프론트엔드와 UI 디자이너가 상태별 버튼/문구/빈 상태를 잡기 위한 기준이다. 백엔드는 이 표를 기준으로 허용되지 않는 상태 변경 요청을 막는다.

### 가용 슬롯 계산 규칙 기본안

가용 슬롯은 유저 앱의 예약 시간 선택 화면과 직접 연결된다. 협업자는 알고리즘을 설계하지 않고, 아래 기본 규칙이 화면 UX에 맞는지 확인한다.

- 기준 시간대: DB는 UTC, 앱/웹 표시는 샵의 로컬 시간대 기준.
- 기본 슬롯 단위: MVP 기본안은 30분 단위. 15분 단위는 팀 내부 결정 필요.
- 슬롯 길이: 디자인의 `duration_minutes`를 기준으로 종료 시간을 계산한다.
- 영업시간 밖 슬롯은 노출하지 않는다.
- 샵 휴무일과 디자이너 휴무/비활성 상태는 제외한다.
- 이미 `pending` 또는 `confirmed` 예약이 겹치는 디자이너는 제외한다.
- `rejected`, `cancelled_by_user`, `cancelled_by_shop` 예약은 슬롯 점유에서 제외한다.
- `no_show`, `completed`는 과거 예약이므로 미래 슬롯 점유에는 영향이 없다.
- 유저가 디자이너를 선택하지 않으면 해당 시간에 가능한 디자이너 중 랜덤 배정한다.
- 가능한 디자이너가 0명이면 해당 슬롯은 예약 불가로 처리한다.
- 당일 예약 마감 시간, 예약 전후 버퍼 시간, 임시 휴무 UI는 팀 내부 결정 필요.

슬롯 조회 응답은 프론트가 화면을 쉽게 만들 수 있도록 다음 정보를 포함하는 것을 기본안으로 한다.

```json
{
  "date": "2026-06-01",
  "timezone": "Asia/Seoul",
  "slot_unit_minutes": 30,
  "slots": [
    {
      "start_datetime": "2026-06-01T05:00:00Z",
      "local_time_label": "14:00",
      "available": true,
      "available_designer_count": 2,
      "available_designer_ids": ["d_1", "d_2"]
    }
  ]
}
```

협업자 확인 사항:

- 시간 선택 화면에서 불가 슬롯을 숨길지, 비활성으로 보여줄지.
- 자동 배정 시 “디자이너 자동 배정” 문구를 어디에 표시할지.
- 가능한 디자이너가 0명일 때 빈 상태 문구.
- 당일 예약 마감 기준이 필요하면 몇 시간 전으로 할지.
- 예약 전후 버퍼 시간이 있으면 사장님 웹에서 어디서 설정할지.

### 샵/디자이너

- 사장님 계정은 이메일, 비밀번호, 대표자명, 카카오 알림톡 수신용 전화번호, 사업자등록번호, 사업자등록증을 가진다.
- 샵은 이름, 주소, 좌표, 전화, 소개, 대표 이미지, 영업시간, 자동수락 여부, 예약 운영 안내, 평균 별점, 리뷰 수, 찜 수를 가진다.
- 디자이너는 샵에 종속되며 이름, 경력, 직급, 프로필 사진, 전문 태그, 활성 여부를 가진다.
- 디자이너 스케줄은 요일, 시작/종료, 휴게, 휴무 여부로 관리한다.

### 사장님 웹 CRUD API 기본안

사장님 웹 작업자는 DB 구조를 직접 설계하지 않는다. 아래 API가 각 화면에서 충분한지, 입력 UI와 검증 문구가 자연스러운지 확인한다.

#### Owner/Auth

- `POST /owner/auth/register`: 사장님 회원가입.
- `POST /owner/auth/login`: 사장님 로그인.
- `POST /owner/auth/logout`: 로그아웃.
- `GET /owner/me`: 내 계정/사업자 인증 상태 조회.
- `PATCH /owner/me`: 대표자명, 연락처 등 계정 정보 수정.
- `POST /owner/business-verification`: 사업자 정보/사업자등록증 제출.
- 협업자 확인 사항: 사업자 승인 전 접근 가능한 메뉴, 승인 대기/반려 화면 문구.

#### Shop

- `POST /owner/shops`: 샵 생성.
- `GET /owner/shops`: 내가 관리하는 샵 목록.
- `GET /owner/shops/{shop_id}`: 샵 상세.
- `PATCH /owner/shops/{shop_id}`: 샵 기본 정보 수정.
- `PATCH /owner/shops/{shop_id}/business-hours`: 영업시간 수정.
- `PATCH /owner/shops/{shop_id}/reservation-policy`: 자동수락, 예약 운영 안내 수정.
- `PATCH /owner/shops/{shop_id}/images`: 대표 이미지/샵 이미지 수정.
- 협업자 확인 사항: 여러 샵 관리 여부, 예약 운영 안내 입력 UI, 영업시간 휴무 UI.

#### Designer

- `POST /owner/shops/{shop_id}/designers`: 디자이너 추가.
- `GET /owner/shops/{shop_id}/designers`: 디자이너 목록.
- `GET /owner/designers/{designer_id}`: 디자이너 상세.
- `PATCH /owner/designers/{designer_id}`: 디자이너 정보 수정.
- `PATCH /owner/designers/{designer_id}/schedule`: 주간 근무시간 수정.
- `POST /owner/designers/{designer_id}/time-off`: 특정일 휴무/반차/임시 불가 시간 등록.
- `DELETE /owner/designers/{designer_id}`: 퇴사/비활성화 처리.
- 협업자 확인 사항: 삭제 대신 비활성화라는 표현 사용, 예약이 있는 디자이너 수정 제한 문구.

#### Design

- `POST /owner/shops/{shop_id}/designs`: 디자인 등록.
- `GET /owner/shops/{shop_id}/designs`: 디자인 목록.
- `GET /owner/designs/{design_id}`: 디자인 상세/LLM 분석 상태 조회.
- `PATCH /owner/designs/{design_id}`: 디자인 정보 수정.
- `POST /owner/designs/{design_id}/images`: 디자인 이미지 추가.
- `DELETE /owner/designs/{design_id}/images/{image_id}`: 디자인 이미지 삭제.
- `POST /owner/designs/{design_id}/reanalyze`: LLM 재분석 요청.
- `PATCH /owner/designs/{design_id}/visibility`: 노출/숨김 변경.
- `DELETE /owner/designs/{design_id}`: 디자인 삭제 또는 숨김 처리.
- 협업자 확인 사항: LLM 분석 중/실패/완료 상태 UI, 삭제와 숨김을 어떻게 구분할지.

### 디자인

- 디자인은 사장님이 등록하는 예약 가능한 상품이다.
- 필수 입력: 제목, 가격, 시술 시간, 가능 디자이너 1명 이상, 이미지 1장 이상.
- 사장님 태그와 AI 태그는 검색에 함께 사용한다.
- 상태 값: `draft`, `transforming`, `classifying`, `active`, `failed`, `hidden`.
- 이미지 원본은 저장하고, LLM 1단계 결과인 규격화 이미지 URL을 노출용으로 사용한다.
- 이미지 업로드 제안: jpg/jpeg/png/heic, 장당 10MB, 긴 변 1080px 이상 권장, 디자인당 1~10장.

### 커뮤니티 스네일

- 통합 커뮤니티 하나만 둔다.
- 앱 명칭은 `스네일`로 사용하고, 내부 엔티티명은 `Snap`을 유지한다.
- 스네일은 리뷰가 아니라 스타일 공유 커뮤니티다. 무신사 SNAP처럼 본인의 네일 사진과 취향을 공유하는 목적이다.
- 커뮤니티 탭 상단은 [스네일] [랭킹] [팔로잉] 3개로 구성한다.
- 스네일은 누구나 작성 가능하며 예약 인증은 필수가 아니다.
- 스네일에는 별점이 없다.
- 스네일은 샵, 디자인, 디자이너, 예약을 선택적으로 태그할 수 있다.
- `tagged_design_id`가 있으면 디자인 상세의 스네일 목록에 노출한다.
- 예약 태그가 본인의 완료 예약이면 인증 뱃지를 줄 수 있다.
- 스네일 좋아요/댓글/조회수/랭킹은 커뮤니티 지표로만 사용하고 샵 평점에는 반영하지 않는다.
- 디자인 상세에서는 “이 디자인을 받은 스네일”과 “예약 기반 리뷰”를 분리해서 보여준다.

### 댓글/좋아요/팔로우

- 댓글과 대댓글은 유저/사장님 모두 작성 가능하다.
- 댓글 depth는 2까지만 허용한다. 대대댓글은 불가하다.
- 댓글 작성자는 `author_type=user|shop`으로 구분하고, 샵 댓글은 뱃지 표시가 가능하다.
- 스네일 좋아요, 댓글 좋아요, 유저 팔로우를 지원한다.

### 리뷰

- 리뷰는 우리 앱 예약 완료 후 작성 가능하다.
- `reservation_id`에 UNIQUE 제약을 둬 1예약당 최대 1리뷰만 허용한다.
- 별점은 샵에 대한 별점만 저장한다.
- 디자이너 개별 별점은 없다.
- 사진은 0~5장이다.
- 리뷰 작성은 선택 사항이다.
- 리뷰 작성/수정/삭제 시 샵 평균 별점과 리뷰 수를 재계산한다.
- 디자인 상세에서는 해당 디자인을 받은 사람들의 리뷰를 보여준다.
- 리뷰는 스타일 공유가 아니라 예약 기반 평가 데이터다.
- 리뷰는 샵/디자인 예약 판단에 도움이 되는 신뢰 정보로 사용한다.
- 리뷰에는 별점이 있고, 스네일에는 별점이 없다.
- 리뷰는 샵 평균 별점에 반영되고, 스네일은 샵 평균 별점에 반영되지 않는다.
- 디자인 상세에서 리뷰와 스네일은 별도 섹션으로 분리한다.

### 신고/모더레이션

- 신고 대상: 스네일, 댓글, 리뷰, 유저, 샵.
- 신고 사유 예시: `spam`, `abuse`, `sexual`, `copyright`, `other`.
- 처리 상태: `pending`, `reviewing`, `resolved`.
- 처리 액션: `none`, `hide`, `delete`, `ban`.

## 인증/권한 모델

권한 모델은 협업자가 “이 버튼을 누구에게 보여줄지” 판단하기 위한 기준이다. 백엔드는 이 표를 기준으로 서버에서 한 번 더 권한을 검증한다.

### 권한 주체

- 비로그인 유저: 검색/피드/상세 일부만 볼 수 있다.
- 로그인 유저: 예약, 찜, 스네일, 댓글, 리뷰를 사용할 수 있다.
- 사장님 Owner: 본인 소유 샵의 정보, 디자이너, 디자인, 예약만 관리할 수 있다.
- 어드민: 신고 처리, 사업자 승인, 콘텐츠 숨김/삭제를 처리한다.
- 시스템: 스케줄러, LLM callback, 알림 발송 등 서버 내부 작업을 수행한다.

### 권한 표

| 기능 | 비로그인 | 로그인 유저 | 사장님 Owner | 어드민 | 협업자 확인 |
| --- | --- | --- | --- | --- | --- |
| 검색/디자인 상세 보기 | 가능 | 가능 | 가능 | 가능 | 비로그인 허용 범위 |
| 예약 생성 | 불가 | 가능 | 불가 | 불가 | 로그인 유도 화면 |
| 내 예약 조회 | 불가 | 본인만 | 불가 | 가능 | 타인 예약 접근 차단 |
| 예약 수락/거절 | 불가 | 불가 | 본인 샵 예약만 | 가능 | 사장님 웹 버튼 노출 |
| 리뷰 작성 | 불가 | 본인 completed 예약만 | 불가 | 불가 | 리뷰 작성 가능 조건 |
| 스네일 작성 | 불가 | 가능 | 불가 | 불가 | 로그인 유도 화면 |
| 댓글 작성 | 불가 | 가능 | 본인 샵 계정으로 가능 | 가능 | 샵 뱃지 표시 |
| 샵 정보 수정 | 불가 | 불가 | 본인 샵만 | 가능 | 소유권 오류 문구 |
| 디자이너/디자인 관리 | 불가 | 불가 | 본인 샵만 | 가능 | 예약 존재 시 제한 문구 |
| 신고 처리 | 불가 | 불가 | 불가 | 가능 | 어드민 MVP 포함 여부 |

공통 권한 오류는 `403 FORBIDDEN`으로 처리한다. 로그인 자체가 필요한 경우는 `401 UNAUTHORIZED`로 처리한다.

## LLM 명세

### 1단계 Transform

- 목적: 원본 이미지에서 네일 영역을 추출하고 규격화된 이미지를 만든다.
- 입력: `image_url`, `image_id`, 선택적 `callback_url`, `options.output_size`, `options.background`.
- 출력 성공: `image_id`, `status=success`, `cropped_image_url`, `cropped_image_size`, `confidence`, `nail_count_detected`, `processing_time_ms`.
- 출력 실패: `status`, `error_code`, `error_message`.
- 5초 이내면 동기 가능, 10초 이상이면 비동기 webhook 권장.
- cropped 이미지 호스팅 책임은 LLM 자체 호스팅 또는 S3 URL 응답 중 결정 필요.

### 2단계 Classification

- 목적: 1단계 결과 이미지를 기반으로 태그, 색상, 스타일, 네일 모양을 분류한다.
- 입력: `image_url`, `image_id`, `locale`, `options.max_tags`, `options.include_color_palette`, `options.include_style_category`.
- 출력: `tags[]`, `color_palette[]`, `style_category`, `nail_shape`, `confidence_overall`, `processing_time_ms`.
- 태그는 표준 태그 사전 내 값만 출력해야 한다.

### 표준 태그 제안

- `style`: 프렌치, 옴브레, 그라데이션, 마그넷, 글리터, 큐빅, 라인아트, 캐릭터, 무광, 유광.
- `color`: 핑크, 레드, 누드, 블랙, 화이트, 베이지, 블루, 그린, 옐로우, 퍼플, 브라운, 골드, 실버.
- `mood`: 봄, 여름, 가을, 겨울, 시크, 러블리, 심플, 글래머, 내추럴, 키치, 모던.
- `technique`: 젤, 매니큐어, 페디큐어, 연장, 케어, 제거.
- `shape`: 스퀘어, 라운드, 오벌, 아몬드, 스틸레토, 발레리나.
- `style_category`: simple, glamour, classic, trendy, chic.
- `occasion`: 웨딩, 데일리, 파티, 오피스, 데이트.

### LLM 에러 코드

- `NO_NAIL`: 사진에서 네일이 감지되지 않음.
- `LOW_QUALITY`: 해상도 낮음, 블러, 조명 부족.
- `MULTIPLE_HANDS`: 여러 명의 손이 섞임.
- `OBSTRUCTED`: 네일이 가려짐.
- `INAPPROPRIATE`: 부적절 콘텐츠 감지.
- `INTERNAL_ERROR`: LLM 내부 에러.

### LLM 작업자 작성 필요 질문

이 영역은 LLM 작업자가 채워야 한다. 백엔드는 아래 질문에 대한 답을 받아 API 호출, 재시도, 저장, 장애 처리를 구현한다.

- Transform/Classify는 동기 응답인가, 비동기 callback인가.
- 비동기라면 callback URL 호출 시 인증 방식은 무엇인가.
- 평균 처리 시간과 p95 처리 시간은 어느 정도인가.
- 백엔드 timeout은 몇 초로 잡아야 하는가.
- 재시도 가능한 에러와 재시도하면 안 되는 에러는 무엇인가.
- rate limit 초과 시 어떤 HTTP status와 body를 반환하는가.
- cropped 이미지는 LLM이 저장하는가, 백엔드 S3에 업로드하는가.
- LLM이 제공하는 이미지 URL의 만료 시간은 얼마인가.
- 응답에 `model_version`을 포함할 수 있는가.
- 모델이 업데이트되면 기존 이미지를 재분석해야 하는가.
- 디자인에 이미지가 여러 장일 때 태그 병합은 LLM이 하는가, 백엔드가 하는가.
- 태그는 표준 태그 사전 값만 반환할 수 있는가.
- confidence가 낮을 때 백엔드가 실패로 처리해야 하는 기준값은 얼마인가.

## 공통 API 협업 규칙

이 섹션은 프론트엔드 작업자가 모든 API를 같은 방식으로 처리할 수 있게 하는 약속이다. 세부 구현은 백엔드가 맡고, 프론트엔드는 이 형태로 처리 가능한지 확인한다.

### 공통 성공 응답

단건 응답은 다음 형태를 기본으로 한다.

```json
{
  "data": { "...": "..." },
  "request_id": "req_abc123"
}
```

리스트 응답은 다음 형태를 기본으로 한다.

```json
{
  "data": [{ "...": "..." }],
  "page": {
    "next_cursor": "opaque_cursor_string",
    "has_next": true
  },
  "request_id": "req_abc123"
}
```

### 페이지네이션

- 리스트 API는 cursor 기반 페이지네이션을 기본으로 한다.
- `cursor`는 프론트가 해석하지 않는 opaque string이다.
- 첫 요청은 `cursor` 없이 보낸다.
- 다음 페이지가 없으면 `next_cursor=null`, `has_next=false`.
- 기본 `limit`은 20, 최대 `limit`은 50을 제안한다.

### 공통 에러 응답

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "입력값을 확인해주세요.",
    "field_errors": {
      "nickname": "닉네임은 2자 이상이어야 합니다."
    }
  },
  "request_id": "req_abc123"
}
```

기본 에러 코드:

- `UNAUTHORIZED`: 로그인이 필요함.
- `FORBIDDEN`: 권한 없음.
- `NOT_FOUND`: 리소스 없음.
- `VALIDATION_ERROR`: 입력값 오류.
- `CONFLICT`: 예약 충돌, 중복 리뷰, 중복 요청 등.
- `RATE_LIMITED`: 요청 제한 초과.
- `INTERNAL_ERROR`: 서버 오류.

### Rate Limit

- MVP 기본안: 로그인 유저 기준 일반 API 분당 60회, 검색 API 분당 30회, 업로드/예약 생성 계열은 더 낮게 제한.
- 제한 초과 시 HTTP 429와 `RATE_LIMITED`를 반환한다.
- 프론트 확인 사항: 제한 초과 문구와 재시도 버튼/대기 안내.

### 시간 포맷

- API 시간은 ISO 8601 문자열을 사용한다.
- 서버 저장 기준은 UTC다.
- 화면 표시는 앱/웹에서 샵 로컬 시간대로 변환한다.

### 빈 상태

- 빈 리스트도 HTTP 200으로 응답한다.
- 검색 0건은 완전 빈 배열 대신 유사 추천 디자인을 포함할 수 있다.
- 프론트 확인 사항: 빈 결과, 유사 추천, 필터 초기화 버튼 문구.

- `GET /shops/{id}`: 샵 상세.

## 확정 의사결정

- 통합 커뮤니티 1개를 둔다.
- 디자인은 사장님 등록 상품, 스네일은 유저 자유 공유로 분리한다.
- 댓글/대댓글은 누구나 가능하고 depth 2까지만 허용한다.
- 리뷰는 1예약 1리뷰, 샵 별점만 저장한다.
- MVP 앱에서는 결제 없이 예약 요청/수락 구조를 사용한다.
- 디자이너 자동 배정은 가능한 디자이너 중 랜덤이다.
- 사장님 알림은 카카오 알림톡이다.
- 검색 결과 0건 시 유사 디자인을 추천한다.
- 이미지 원본은 LLM 재처리를 위해 보관한다.
- soft delete 후 30일 뒤 hard delete를 고려한다.
- DB 시간은 UTC, API 시간은 ISO 8601, API 버전은 `/v1`을 사용한다.
