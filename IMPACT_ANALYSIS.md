# 🎯 Impact Analysis 워크플로우

> 기능 수정·추가 요청이 들어왔을 때 **어떤 구조(결정 매트릭스 / 모델 / 스키마 / API / 마이그레이션)를 바꿔야 하는지 빠르고 정확하게 알아내는** 표준 절차.
>
> 이번 IA v2 화면명세 검토(2026-05-27)의 경로를 거꾸로 정리한 것. 다음 요청도 이 순서로 한다.

---

## 0. 언제 이 문서를 쓰는가

| 입력 유형 | 예 |
|---|---|
| **외부 요청** | 프론트 화면 명세, 디자이너 시안, 팀장 기능 요청, 사용자/QA 피드백 |
| **내부 제안** | 내가 spec/코드 읽다가 "여기 구조 바꾸면 좋겠다" 발견한 경우 |
| **결정 번복 검토** | §11 결정 매트릭스 항목을 다시 열고 싶다는 요청 |

워크플로우 발동 트리거: "**기존 구조와 다른 무언가가 요구되었거나 발견되었을 때**".  
사소한 버그 수정·문구 변경은 이 절차 생략 (codex 미니 패치로 바로).

---

## 1. 핵심 원칙 — 싼 검사부터 멈춘다

```
싼 검사 ────────────────────────────────────► 비싼 검사
§11 결정 매트릭스   →   모델 컬럼   →   스키마/서비스   →   라우터/마이그레이션
(파일 1개)         (파일 1~3개)    (도메인 폴더)      (인덱스·테스트)
```

각 단계에서 **결론 나면 다음 단계 스킵**.  
예: §11에서 충돌 잡히면 모델 안 읽어도 됨 → 사용자 결정부터 받기.

---

## 2. 4단계 체크리스트

### Step 1. §11 결정 매트릭스 충돌 스캔 (필수, 30초)

**읽을 파일**: `AGENTS_PLAN.md §11` 또는 `HANDOFF.md §5`

**물을 것**:
- 이 요청이 확정된 7+α 결정 중 어느 것과 충돌하는가?
- 대기 결정(지도/추천/이달의아트/영업휴게 등) 영역인가?

**출력**:
- 충돌 있음 → 🔴 (사용자 결정 필요, 모델 검사 스킵해도 됨)
- 대기 결정 영역 → 🟡 (피드백 대기 큐에 추가)
- 충돌 없음 → Step 2로

> **트랩**: 사용자가 한 번 확정한 결정을 spec_canonical 따라가서 번복하지 말 것. SSOT는 §11.

---

### Step 2. 모델/컬럼/FK 존재 확인 (1~2분)

**읽을 파일**: `backend/app/models/{도메인}.py` (최대 3개)

**물을 것**:
- 화면이 요구하는 필드가 모델에 있는가? (예: `users.mode_preference`, `shops.latitude`)
- FK·인덱스 관계가 동선을 지원하는가? (예: `Snap.tagged_design_id`)
- 새 테이블이 필요한가? (예: `FavoriteShop`, `FavoriteFolder`)

**출력**:
- 모든 필드 존재 → Step 3으로
- 컬럼 1~2개 누락 → 🔵 **모델 미니 패치** (마이그레이션 1개)
- 새 테이블 필요 → 🔵 **모델 추가** + 사용자 확인 (MVP 범위?)

---

### Step 3. 스키마/서비스/라우터 표면 확인 (2~5분)

**읽을 파일**: `backend/app/schemas/`, `backend/app/services/`, `backend/app/api/v1/`

**도구**: Grep으로 `tagged_design_id|design_id|필드명` 빠른 검색

**물을 것**:
- 필요한 엔드포인트가 있는가? (URL + 메서드)
- 필터 파라미터가 있는가? (예: `SnapFeedQuery.tagged_design_id`)
- 응답 스키마에 필드가 노출되는가? (예: `DesignPublic`에서 `ai_tags` 제외 여부)
- 비즈니스 로직이 제약을 강제하는가? (예: 리뷰 7일 내 수정 제한)

**출력**:
- 표면이 부족함 → 🔵 **API 보완 패치** (schema 필드 + service 분기 + router)
- 응답 스키마 노출만 조정 → 🔵 **스키마 미니 패치** (1~2줄)
- 다 됨 → 🟢 (이미 지원, 프론트 안내만)

---

### Step 4. 마이그레이션·인덱스·테스트 영향 (필요 시)

**읽을 파일**: `backend/alembic/versions/`, `backend/tests/`

**물을 것**:
- 새 FK 컬럼에 인덱스 필요한가? (FK 자동 인덱스는 PostgreSQL 기본 X)
- 노출 가드 4종 영향받는가? (verification_status / visibility / ai_analysis_status / deleted_at)
- 통합 테스트 새로 필요한가?

**출력**:
- 새 마이그레이션 N개, 테스트 케이스 N개로 정리해서 codex 미니 패치 프롬프트에 넣음

---

## 3. 5분류 출력 템플릿 (보고 형식)

모든 검토 결과는 이 분류로 정리한다:

```markdown
## 🔴 결정 매트릭스 직접 충돌 (사용자 판단 필요)
- [#번호] 항목 + 어느 결정과 충돌 + 선택지 A/B/C
  → AskUserQuestion 발동

## 🟡 대기 결정과 겹침
- 항목 + 어느 대기 결정 영역인지 + 영향도(A5에 영향 있음/없음)
  → 피드백 대기 큐에 메모, A5 진행은 막지 않음

## 🔵 백엔드 추가 작업 필요
| 화면/요청 | 누락 | 작업량 |
|---|---|---|
| ... | 모델 컬럼 N / 마이그레이션 N / API N / 인덱스 N | S/M/L |
  → A5 이후 미니 패치 묶음 또는 즉시 처리

## 🟢 이미 지원 (변경 불필요)
- ... → 프론트에 "현재 구조로 가능" 안내

## ⚪ 프론트 only 또는 MVP 외
- 프론트에 컴포넌트만 만들면 되는 것
- MVP 제외 권장 항목 (쿠폰/초대코드 등)
```

---

## 4. 후속 액션 매핑

| 분류 | 액션 |
|---|---|
| 🔴 | `AskUserQuestion`으로 선택지 A/B/C 제시 → 답변받으면 §11에 결정 추가 → `AGENTS_PLAN.md` + `HANDOFF.md` 갱신 |
| 🟡 | 피드백 대기 큐에 1줄 추가 (HANDOFF.md §5 하단). A5 등 진행에 영향 없으면 계속 진행 |
| 🔵 | codex 미니 패치 프롬프트 작성 (스코프, 변경 파일, DoD 명시) — 사용자가 codex로 실행 |
| 🟢 | 프론트 회신 메시지에 "OK, 현재 구조로 가능" 명시 |
| ⚪ | 프론트 회신에 "프론트 단독" 또는 "MVP 제외 권장" 명시 |

---

## 5. 토큰 절감 트릭

1. **Memory 먼저** — `MEMORY.md` + project_snail_status 읽으면 현재 진행 상태 즉시 파악
2. **HANDOFF.md §5** = §11 결정 요약. 전체 AGENTS_PLAN 안 읽고 이 표 한 장만 봐도 충돌 잡힘
3. **모델 파일은 도메인별로만** — Auth 요청이면 accounts.py만, Shop 요청이면 shop.py만
4. **Grep > Read** — 컬럼·필드 존재 여부는 Grep으로 30초, 전체 Read는 비쌈
5. **다 끝나고 한 번에 정리** — 검토 중간에 요약 X. 마지막에 5분류 표로 한 번에

## 6. 실 사례 — IA v2 화면 명세 검토 (2026-05-27)

입력: 프론트가 작성한 화면 명세 v2 (44개 화면 + 7개 모달)

소요 시간: 약 10분 (모델 3개 + 라우터 1개 읽음, 나머지는 Grep)

결과:
- 🔴 3건 → AskUserQuestion → 전부 권장안 채택 → §11에 결정 #8 추가, #2/#6 명확화
- 🟡 4건 → 대기 결정 큐에 보존
- 🔵 11건 → A5 이후 미니 패치 묶음 후보 (FavoriteShop, mode_preference, 검색 자동완성 등)
- 🟢 약 10건 → 이미 지원 확인
- ⚪ 1건 → 쿠폰/초대코드는 MVP 제외 권장

산출물:
- `AGENTS_PLAN.md §11` 갱신
- `HANDOFF.md §5` 갱신
- 프론트 회신 변경 요청 5건 정리

> 다음에 같은 규모 요청 들어오면 이 케이스가 기준선. 더 길어지면 검토 방식 점검 필요.

## 7. 입력별 최단 경로 (빠른 참조)

| 입력 유형 | 1순위 확인 | 흔히 막히는 곳 |
|---|---|---|
| 화면 명세 변경 | §11 충돌 → 응답 스키마 노출 → 새 엔드포인트 필요성 | 비기능 요구(차단/모더레이션/카테고리 토글)가 결정 #3, #5, #6과 부딪힘 |
| 신규 데이터 노출 | 모델 컬럼 존재 → 스키마 필드 노출 여부 | AI 태그 같은 노출 정책 (DesignPublic) |
| 신규 동선/이동 | FK 존재 → 필터 파라미터 또는 신규 라우터 | tagged_* FK는 있어도 GET 필터가 없는 경우 많음 |
| 새 비즈니스 규칙 | reservation_policy.py 등 정책 모듈 | 상수 변경만으로 끝나는지 vs 모델 컬럼 추가 필요한지 |
| 결정 번복 요청 | §11 + 영향받는 에이전트 코드 영역 | 번복 비용(마이그레이션 + 기존 데이터)을 사용자에게 명확히 |
| 내부 발견(개선 제안) | 일단 §11과 충돌 안 하는지 → 사용자 결정 받기 | 본인 판단으로 진행 금지. 사용자 결정 매트릭스에 등재 |

## 8. 이 문서 자체의 유지보수

- 새 결정 매트릭스 항목 추가 시 §7 표 업데이트
- 검토 1회당 소요 시간이 위 사례(~10분)보다 3배 이상 길어지면 → 절차 자체 점검
- §11 결정이 10개 넘어가면 카테고리화 (현재 7+α는 표 그대로 OK)
